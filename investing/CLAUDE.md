# Investment workspace (swing-first)

A Shariah-gated decision-support system for **short-horizon, high-asymmetry swing
trades**. It gates trade **quality** (setup completeness, honest asymmetry, a gap
plan, verified compliance); it never caps trade **quantity** or sizing — those are
the owner's decisions. Each file in `holdings/` is one open position; each file in
`setups/` is one pre-trade plan.

## The loop

`discover.py → leads.md → setups/<ticker>.md → Zoya/Musaffa → recommend.py → owner sizes & executes → /apply-trade → journal.py`

Discovery is machine-wide and writes **`leads.md`** (never `watchlist.md`, which is
hand-curated). A lead is promoted only when the owner writes a setup card and
screens the name compliant. `recommend.py` then re-gates it.

## Decision engine

The engine uses the PM-grade framework from `PM_FRAMEWORK.md` — a buy-side
decision process built on four pillars: a falsifiable edge, variant perception,
reward:risk skew, and conviction-sized positions. For a **swing** trade the "edge"
is a completed **setup card** (a defined, repeatable setup), not a fundamental thesis.

**Seven labels, two domains:**
- Existing holdings → SELL > TRIM > REVIEW > HOLD (precedence order)
- New ideas → BUY-CANDIDATE | RESEARCH | AVOID ; discovery output is only ever **LEAD**

**Gate order (runs in sequence):**
1. Shariah knockout — non-compliant / ratio-or-business flag → AVOID/SELL, absolute
2. Liquidity floor (discovery) — under `screen_min_mcap` / `screen_min_avg_dollar_vol` → dropped
3. Edge test — no completed `setups/<ticker>.md` card → cap at RESEARCH (LEAD from discovery)
4. Asymmetry gate — reward:risk < `reward_risk_min_swing` → cannot be BUY-CANDIDATE
5. Catalyst-within-horizon gate — no catalyst inside the window → RESEARCH / DEAD_MONEY
6. Shariah verified on the card — `compliant` + fresh; UNVERIFIED can never be BUY-CANDIDATE
7. Gap plan — earnings in the holding window with no valid `earnings_plan` → GAP_PLAN_MISSING (blocks)
8. Sizing — `risk_per_trade_pct` to the stop; gap-adjusted if holding through a print; `max_position_pct` cap

## How to run the routines

| Slash command | What it does |
|---|---|
| `/discover` | Build `leads.md` from a machine-wide pool (liquidity + Shariah/asymmetry/catalyst gates) |
| `/assess-portfolio` | Full assessment: holding verdicts, PM records, catalyst research, leads |
| `/screen-ideas` | Rank `universe.md` by signals, then apply the setup/asymmetry/catalyst/Shariah gates |
| `/apply-trade` | Record an executed trade; logs entry+exit to `journal.csv`; prompts for card fields on new buys |

## Key files

| File | Purpose |
|---|---|
| `PM_FRAMEWORK.md` | Full framework reference (citations, schema, label definitions) |
| `rules.md` | Tunable knobs (YAML) + decision rules prose — edit here to change policy |
| `holdings/*.md` | One open position per file; YAML front-matter + thesis body |
| `setups/*.md` | One pre-trade setup card per prospective trade (`_template.md`, `README.md`) |
| `leads.md` | MACHINE-generated by `discover.py` (overwritten each run) — do not hand-edit |
| `watchlist.md` | HAND-curated candidate list — no script writes it |
| `scripts/discover.py` | Auto-discovers leads; liquidity floors; writes `leads.md` |
| `scripts/recommend.py` | New-idea funnel (BUY-CANDIDATE/RESEARCH/AVOID) + gap-aware sizing |
| `scripts/verdict.py` | Mechanically resolves SELL/TRIM/REVIEW/HOLD per holding |
| `scripts/journal.py` | Per-setup expectancy report from `journal.csv` |
| `scripts/apply_txn.py` | Records executed trades; writes `transactions.csv` + `journal.csv` |
| `reports/` | Dated assessment outputs |

## Holdings front-matter schema

Scripts parse YAML between `---` markers. Key PM-grade fields the user fills in
themselves at entry (the engine never invents these):
- `conviction`, `thesis_one_liner`, `variant_view` — qualitative edge/conviction
- `catalyst` (type/desc/date) — hard or soft, when it closes the price/value gap
- `initial_stop` — price-based defensive level
- `target_price` / `target_method` — the level your method points to
- `invalidation` — thesis-type trigger that proves you wrong (distinct from stop)
- `pre_mortem` — top failure reasons written at entry, before emotion revises them
- `last_review` — date of the last "would I buy this here today?" re-underwrite

## Conventions
- Tadawul tickers use the `.SR` suffix (e.g. `2222.SR`). US tickers as-is.
- Money is for the user's own analysis — never frame output as financial advice.
- The Shariah ratio pre-check is a heads-up only; Zoya/Musaffa are the source of truth.
- Don't fabricate figures: if a data fetch fails, report the gap.
